# Loupe — PRD

**한 줄 소개:** _내가 모르는 분야 전문가를, 내 기준으로 X에서 발견한다._

특정 토픽에 대해 **X에 있는 진짜 전문가를 발견**하는 도구. 토픽은 사용자가 정하고, AI가 평가 기준을 제안하고, 사용자가 그 기준을 확정하면, Loupe가 시드+네트워크 확장으로 후보를 모으고 다중 소스로 검증해서 점수 매긴다. 부가적으로 "내가 이미 팔로우 중인 사람들이 그 기준에 얼마나 부합하는지"도 같은 도구로 보여준다.

> 참고: 앱 UI는 영어, 이 PRD만 한국어.

---

## 1. 배경

이 앱은 이전 프로젝트인 **Atrium**의 후속작이다. Atrium은 핵심 가설 — _**신뢰는 사람 단위가 아니라 (사람, 토픽) 단위다**_ — 을 증명했지만, 두 가지 한계가 있었다.

1. **기준을 저희가 정했다.** AI / Startup-VC 두 도메인에 고정 기준을 박았다. 감사 가능성은 좋지만 주관적이고 두 도메인에 갇혀 있다.
2. **새로운 사람 발견을 못 했다.** Atrium은 큐레이션된 작은 풀에서만 추천했다. 사용자가 모르는 분야의 새 전문가를 X에서 찾아주지 못했다.

**Loupe는 둘 다 뒤집는다.** 기준은 사용자가 정의하고, 발견 범위는 X 전체로 확장한다.

---

## 2. 풀고자 하는 문제

내가 새롭게 진지해진 분야가 있다. 그 분야의 **진짜 전문가가 X에 분명히 있는데, 누구인지 모른다.**

지금 X에서 토픽을 검색하면:
- **인기도순으로만 정렬** → 바이럴된 평범한 글이 위에
- **이미 유명한 사람** 위주 → 조용하지만 깊이 있는 사람은 안 보임
- **내 기존 네트워크 안**에서만 추천 → 외부의 전문가는 영영 못 만남
- **"전문성"이라는 정렬 기준 자체가 없음** — 좋아요 수와 전문성은 다른 차원

기존 솔루션도 모두 부족:
- **고정 큐레이션 (Atrium, Substack)** — 작은 풀, 외부 발견 X
- **인기도 기반 정렬** — 바이럴이 전문성을 이김
- **수동 리스트 관리** — 사용자가 직접 분류해야

Loupe는 이 질문에 답한다 — **"이 토픽에서, 내가 정한 기준에 따라, X에 있는 진짜 전문가가 누구인가?"**

부가 기능: **"내가 이미 팔로우 중인 사람들 중 이 기준에 부합하는 사람은 누구인가?"**

---

## 3. 핵심 아이디어

세 가지 동작:

### ① 사용자가 토픽을 명명한다
자유 텍스트. _"AI product design"_, _"Korean indie folk songwriting"_, _"Rust async runtimes"_ 등. 옵션으로 서브카테고리 힌트.

### ② AI가 평가 기준을 제안하고, **사용자가 소유한다**
AI가 5개 기준을 가중치와 통과 예시 계정 3명씩과 함께 생성. 사용자가 검토/수정/추가 후 확정. 이 출력물은 **사용자의 개인적인 "이 토픽 전문성 정의"**다.

기준 확정 직전 **캘리브레이션 테스트**: 사용자가 본인의 "gold-standard" 전문가 한 명을 입력 → 그 계정이 이 기준에서 4/5 이상 통과하는지 즉석 검증 → 통과 못 하면 기준 재생성 제안.

### ③ Loupe가 두 가지를 동시에 보여준다
- **🔍 Discover** — 시드 계정에서 출발해 네트워크를 1-2 hop 확장, 후보 풀을 모은 뒤 다중 소스 검증으로 점수화. **메인 기능.**
- **🪞 My Follows** — 사용자가 X 핸들 입력하면, 그가 팔로우하는 사람들을 같은 기준으로 점수화. **보조 기능.**

①과 ②가 새로운 부분, ③이 실제로 유용한 부분.

## 4. 사용자 흐름

### 화면 1 — 토픽 정의
- 단일 텍스트 입력: _"What expertise do you want to find on X?"_
- 옵션: 서브카테고리 힌트
- CTA: **Generate criteria**

### 화면 2 — 기준 검토
- AI가 **5개 기준** 카드로 반환:
  - 라벨, 설명, 가중치 슬라이더
  - 통과할 만한 X 계정 예시 3명
- 사용자: 언체크 / 수정 / 추가 (최대 7개) / 가중치 조정
- 화면 하단 **캘리브레이션 테스트**:
  - _"Test this rubric — name your gold-standard expert"_
  - 핸들 입력 → 즉석 점수 → 4/5 미만이면 재생성 제안
- CTA: **Confirm rubric**

### 화면 3 — 모드 선택 + 핸들 입력
- 두 카드: 🔍 Discover (기본 체크) / 🪞 Score people I follow
- 사용자 X 핸들 입력 (Score 모드 선택 시 필수)
- CTA: **Find experts**
- 화면이 진행 표시기로 전환:
  - _"Seeding from rubric examples (9 accounts)..."_
  - _"Expanding network (47 candidates)..."_
  - _"Verifying credentials..."_
  - _"Scoring against rubric..."_

### 화면 4 — 결과
탭: **🔍 Discover (default)** / **🪞 My Follows**

각 탭 안에서:
- 적합도 순으로 정렬된 계정 리스트, 0–100 점수
- 각 행:
  - 아바타, 핸들, 표시 이름, 적합도 점수
  - 통과한 기준 pill 리스트 + **검증 강도 점** (🟢/🟡/🔴)
  - 펼치면: 기준별 분석 + 근거 트윗 인용 + 외부 검증 링크
- 필터: top 20 / pass ≥ 3 criteria / 검증 강도 ≥ 🟡 등
- CTA: **Open on X** (계정별), **Follow on X** (편의 링크)

### 화면 5 (스트레치) — 저장된 rubric 재실행
- "이 기준 저장하고 매주 재실행" 옵션. MVP에선 빼지만 데이터 모델은 호환되게.

## 5. 핵심 메커니즘

### 5.1 기준 생성 (LLM 호출 #1) — _80% 납득률을 위한 6가지 기법_

**6가지 기법을 합쳐서 첫 출력 품질을 끌어올린다.**

#### (A) 2단계 생성 + Extended Thinking
- **1단계 — Domain archetype 식별**: AI가 먼저 "이 토픽의 전문성은 어떤 모양인가"를 사고. 학술적/산업적/craft/community/hybrid 중 무엇인지.
- **2단계 — archetype에 맞춰 기준 생성**: 가중치 분배도 archetype 별로.
- Claude Sonnet 4.6의 **extended thinking** 활용.

#### (B) 다양한 도메인의 Few-shot
시스템 프롬프트에 정답지 5개 박음 — 학술 AI / K-pop 팬덤 / indie 게임 개발 / 임상 면역학 / 스탠드업 코미디. 각 예시: 토픽 → archetype → 5개 기준 + 가중치 + 이유. 모델한테 "정당한 기준의 모양 범위"를 캘리브레이션.

#### (C) 명시적 anti-pattern 차단
프롬프트에 "절대 만들지 마" 리스트:
- ❌ 동어반복 ("Knows about X")
- ❌ 모든 토픽에 똑같이 쓸 수 있는 일반론
- ❌ 검증 불가 ("Smart thinker")
- ❌ 도메인 미스매치 (K-pop에 박사학위)
- ✅ 공개 X 활동/웹에서 관찰 가능
- ✅ 통과/탈락이 객관적

#### (D) 예시 명명 가능성 강제
**핵심 규칙**: 각 기준마다 명백히 통과하는 실제 X 계정 3명을 명명할 수 있어야 함. 명명 못 하면 그 기준은 너무 모호 — 재생성.

#### (E) Self-critique 패스
초안 후 AI가 자기 비평: 동어반복? 진짜 전문가가 3개 미만 통과할 가능성? 도메인 내부자가 봐도 정당? 문제 발견하면 그 항목만 재생성.

#### (F) UX 캘리브레이션 테스트
화면 2 하단에서 사용자한테 "gold-standard 전문가 1명 명명해보세요" → 즉석 점수 → 4/5 미만이면 _"기준이 어긋난 것 같습니다. 재생성?"_ 제안.

**왜 이게 강력한가**: 추상적 검증("이 기준 맞나?") → 구체적 검증("내 영웅이 통과하나?")로 변환됨. 사용자가 답할 수 있는 질문.

#### 기준 생성 출력 (구조화 JSON)
```json
{
  "domain_archetype": "design + technology hybrid (industry, not academic)",
  "criteria": [
    {
      "id": "kebab-case-id",
      "label": "Human label",
      "description": "1-2 문장",
      "weight": 25,
      "examples": ["@handle1", "@handle2", "@handle3"],
      "verification_sources": ["twitter", "github", "personal-site"]
    }
    // 총 5개, 가중치 합 100
  ]
}
```

### 5.2 후보 풀 모으기 (Discovery 모드) — _Multi-source Discovery_

후보 풀을 **3개 소스에서 합집합**으로 모음. 시드 네트워크에만 의존하면 시드와 연결 안 된 좋은 전문가를 놓침.

#### Phase 1: 시드 모음
기준 생성 시 AI가 제시한 **예시 계정 3명 × 5기준 = 최대 15명** (중복 제거 후 ~9-12명).
기준 생성 단계에서 AI가 **검색어 5-10개**도 함께 생성 (예: `"AI UX patterns"`, `"shipping AI features"`).

#### Phase 2: 후보 풀 확장 (3개 소스 합집합)

**Source A — 시드 1-hop 네트워크**
- 시드의 최근 트윗에서 자주 인용/리트윗/답글하는 핸들 추출
- 시드들이 공통으로 팔로우하는 핸들 (set intersection)

**Source B — X 키워드 검색** (Apify)
- AI가 생성한 검색어로 트윗 검색
- 결과 트윗의 활발한 저자 top 30-50 추출

**Source C — X 바이오 검색** (Apify)
- 바이오에 토픽 핵심 단어 들어간 계정 top 20-30
- 자칭이지만 풀의 다양성 확보용

→ 합집합 dedupe 후 후보 풀 약 80–150명.

#### Phase 3: 1차 필터링 (LLM 1회 호출)
모든 후보의 **바이오 + 핸들**만 보고 명백히 무관한 사람 제거 (정치인, 셀럽, 비공개, 봇). 풀이 50–80명으로 압축.

이 단계 통과한 사람들만 풀파이프라인 (비용 제어).

### 5.3 다중 소스 검증 — _트위터만으로는 부족_

**원칙**: 트위터는 시작점이지 끝점이 아님. 기준의 종류에 따라 검증 소스가 달라야 함.

| 기준 종류 | 검증 소스 |
|---|---|
| **공개 활동/대화** | 트위터만 |
| **학술 자격, 논문, 인용** | Google Scholar / Semantic Scholar API + ArXiv |
| **제품 출시, 코드 기여** | GitHub API (활동, 스타, 컨트리뷰션) |
| **회사 소속 / 시니어 직책** | LinkedIn (Apify 스크랩) + 회사 about 페이지 fetch |
| **공개 글쓰기** | 바이오 링크 따라가기 (Substack/Medium/개인 블로그) |
| **미디어 인정 / 강연** | Tavily 웹 검색 ("@핸들 + 토픽" + "podcast"/"interview"/"keynote") |

#### 검증 파이프라인 (계정당 5단계)

```
1. Twitter 데이터 수집
   ├─ 바이오 (자칭 직함, 자칭 소속, 링크들)
   ├─ 최근 트윗 30–50개 (Apify)
   └─ 핀 트윗

2. 바이오 링크 따라가기
   └─ 개인 사이트, GitHub, LinkedIn, 회사 페이지 fetch (HTML 파싱)

3. 외부 소스 조회
   ├─ Tavily 웹 검색: "@핸들 + 토픽" 메이저 매체 인용
   └─ 도메인별 특수 검색 (조건부):
      ├─ 학술 archetype → Semantic Scholar API
      ├─ 코드/소프트웨어 archetype → GitHub API
      └─ 공공인 archetype → Wikipedia API

4. Cross-reference (LLM 호출)
   ├─ 바이오 주장 ↔ 외부 검증 일치?
   ├─ 트윗 주장 ↔ 외부 활동 흔적?
   └─ 모순 잡기 ("Anthropic Senior" 자칭하지만 회사 페이지에 없음)

5. 최종 점수화 (LLM 호출, 5개씩 배치)
   └─ 기준별 통과/탈락 + 신뢰도 + 검증 강도 + 근거 인용
```

#### 점수화 출력 (구조화 JSON)

```json
{
  "@handle": {
    "criteria_results": [
      {
        "criterion_id": "shipped-ai-features",
        "passes": true,
        "confidence": 0.85,
        "verification_strength": "verified",  // verified / indirect / claimed
        "evidence_tweet_ids": ["1234", "5678"],
        "external_evidence": [
          {"source": "github", "url": "...", "note": "Maintainer of X-AI library, 2.4k stars"},
          {"source": "personal-site", "url": "...", "note": "Lists 3 shipped products"}
        ],
        "rationale": "Maintainer of widely-used AI UI library; tweet thread #1234 walks through production deployment."
      }
    ]
  }
}
```

#### 검증 강도 표시 (UI)
- 🟢 **Verified** — 외부 소스로 cross-check 됨 (회사 페이지, 학교, GitHub 등)
- 🟡 **Indirect** — 트위터 활동만으로 추론, 외부 증거 부족
- 🔴 **Claimed** — 본인 주장만 있고 외부 증거 없음

→ 사용자가 점수만 보고 맹신하지 않게 **증거 강도까지 노출**.

### 5.4 후보 풀 cap

- **Discovery**: 1차 필터링 후 최대 80명까지 풀파이프라인. 결과는 top 20 노출.
- **My Follows**: 최대 300명까지. 풀파이프라인은 fit score 50 이상 후보에만, 나머진 트위터+바이오 점수만.

비용 예상 (Discovery 80명 풀파이프라인):
- LLM (criteria gen + 80개 점수화): ~$1.50
- Apify (트윗 + 링크 fetch): ~$0.50
- Tavily 검색: ~$0.40
- → **세션당 약 $2.40**

## 6. MVP 범위 외 (Out of scope)

- 진짜 X OAuth (Apify 스크랩으로 대체)
- 앱 안에서 팔로우/언팔로우 (X로 링크 아웃만)
- 영구 사용자 계정 / 기준 저장 (MVP는 단일 세션)
- 토픽 추천 ("내가 뭐에 관심 있어야 하지?")
- 평가된 전문가들 간의 의견 불일치 감지
- 트윗 풀텍스트 검색
- 기준 재실행 알림 ("이번 주 새로 등장한 후보")

## 7. 기술 스택

- **프론트엔드 / 백엔드**: Next.js 16 (App Router) — atrium과 동일 패턴
- **호스팅**: Vercel (atrium과 분리된 새 프로젝트)

- **학술 검색**: Semantic Scholar API (무료)
- **코드 시그널**: GitHub REST API (무료)
- **HTML 파싱**: cheerio (개인 사이트 / 회사 about 페이지)
- **Repo**: 새 GitHub 레포
- **스타일링**: Tailwind, 다크모드 기본
- **언어**: 앱 UI는 영어 (PRD만 한국어)

## 8. 솔직한 한계

1. **비공개 자격증은 검증 못 함**
   학위, 비공개 고용 정보는 외부 페이지에 안 적혀 있으면 자칭(🔴)으로만 처리. UI에 명시.

2. **품질 vs 양**
   "AI 논문 50편" 자체로는 좋은 논문인지 모름. 인용수가 일부 도움. 그래서 가중치는 _"양 + 인용 + 최근성"_ 결합으로.

3. **조용한 전문가 손해**
   진짜 전문가지만 X 활동이 적으면 시드 후보로 안 잡힘. 결과 화면에 명시: _"Loupe scores **public web activity**, not absolute expertise."_

4. **LLM 비결정성**
   같은 계정 두 번 점수 매기면 약간 다를 수 있음. 근거 ID 인용으로 사용자가 직접 검증 가능하게.

5. **사실 검증 X**
   _출처_ 큐레이션이지, 그들의 주장 사실 여부 검증 아님.

6. **"전문가" 단어의 위험성**
   UI에서 _"expert"_ 보다 _"strong fit"_, _"high-signal"_ 같은 절제된 표현 사용. 검증 강도와 함께 노출.

## 10. 성공 기준 (데모)

- 사용자가 임의의 토픽 입력 → 15초 안에 합리적인 기준
- **80% 이상의 청중이 "이 기준 합리적"이라고 동의**
- 캘리브레이션 테스트로 "내 영웅도 통과한다" 즉석 확인 가능
- Discovery 결과에서 청중이 모르는 진짜 전문가 1-2명 등장 → "오, 이 사람 처음 들었는데 진짜 전문가네"
- 검증 강도(🟢🟡🔴) 표시로 _"이 도구는 정직하다"_ 인상

## 12. 구현 단계 (해커톤 일정)

- **Phase 1 (1.5h)**: Next.js 부트스트랩 + 토픽 입력 + 기준 생성 (Claude API + 6가지 기법). 화면 1, 2.
- **Phase 2 (1h)**: 캘리브레이션 테스트 (즉석 점수). 화면 2 하단.
- **Phase 3 (2h)**: Seed-and-expand 후보 풀 + 1차 필터링. 화면 3 진행 UI.
- **Phase 4 (3h)**: 다중 소스 검증 파이프라인 (Apify + Tavily + Scholar + GitHub). 5단계 풀.
- **Phase 5 (2h)**: 결과 화면 — Discovery / My Follows 탭, 검증 강도 표시, 근거 노출. 화면 4.
- **Phase 6 (1h)**: 폴리싱, 비용 표시, 캐시 데모, Vercel 배포.

총 약 **10.5시간**. Phase 4가 가장 길고 핵심. Phase 1 끝나면 핵심 가치(기준 생성 + 사용자 편집)는 보여줄 수 있음.
